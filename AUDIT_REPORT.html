📋 审查总结

本次审核以商业级桌面软件为标准，对 SpecReader AI 的前端、Rust 后端、配置文件、测试体系及商业化基础设施进行了全面审查。整体而言，项目工程素质良好——架构清晰、Tauri 命令与纯逻辑解耦、PDF 渲染与交互细节专业、Rust 端序列化往返测试设计出色。但距离商业级交付标准仍有明显差距，主要集中在安全加固、数据可靠性、测试覆盖、商业化基础设施四个方面。

🔴 Critical 级问题（5 项）

Critical C-1. handleFollowUp 在 setSessions updater 内发起副作用，StrictMode 下流式请求双发
文件：src/hooks/usePersistence.ts:444-457
handleFollowUp 在 setSessions 的 updater 函数内部调用 runSessionStream() 发起 LLM 流式请求。React 18 StrictMode（main.tsx:6 已启用）在开发模式下会双重调用 state updater 以检测副作用。这导致：
① 两条并发流式请求同时消费 LLM API，双倍 token 消耗；
② 两个 AbortController 写入同一 messageId，后者覆盖前者，前者流成为孤儿无法 abort；
③ 两个流同时更新同一 session，产生内容交错和状态不一致。
将副作用移出 updater：先在 updater 中只更新 state 并捕获 updatedSession 引用，updater 返回后再调用 runSessionStream()。
Critical C-2. session_path 路径穿越漏洞，可覆盖任意文件
文件：src-tauri/src/lib.rs:284-287
session_path 将前端传入的 session_id 直接拼入文件名，无任何校验。若 session_id 包含 ../，可逃逸 sessions 目录，覆盖 AppData 下任意文件（如 settings.json）。save_session 和 delete_session 命令均受影响。虽然前端当前生成 UUID，但 Tauri command 参数可被任意前端代码（含 XSS 注入代码）构造。
校验 session_id 仅含字母数字和连字符：session_id.chars().all(|c| c.is_alphanumeric() || c == '-' || c == '_')，或规范化路径后校验仍在 sessions_dir 内。
Critical C-3. 完全无自动更新机制
文件：全项目
无 tauri-plugin-updater 依赖、无 updater 配置、无更新检查逻辑。商业级桌面软件必须具备自动更新能力，否则用户无法获取安全补丁和新功能。PRD 中已规划 License 系统，但连基础的更新通道都未建立。
集成 tauri-plugin-updater，配置更新端点 URL 和签名密钥，在应用启动时检查更新。
Critical C-4. 无代码签名配置，分发将被系统拦截
文件：src-tauri/tauri.conf.json:42
macOS signingIdentity: "-" 仅 ad-hoc 签名，用户打开时会被 Gatekeeper 阻止。Windows 无代码签名配置，exe 会触发 SmartScreen 警告。CD 流水线（.github/workflows/cd.yml）仅构建 Windows 且使用 --no-bundle 跳过打包，缺少 macOS 和 Linux 构建。商业软件无法以此状态分发。
配置 Apple Developer ID 签名 + notarization；添加 Windows 代码签名证书；CD 添加 macOS（含 universal binary）和 Linux 构建_job。
Critical C-5. 无 LICENSE 文件，许可证信息缺失
文件：package.json（无 license 字段）、src-tauri/Cargo.toml:6（license = "" 为空）、项目根目录无 LICENSE 文件
PRD 明确标注"商业软件"，但代码库无任何许可证声明。Cargo.toml 的 description/authors/repository 均为占位符。这在法律和商业分发上都是不可接受的。
添加 LICENSE 文件（商业许可或选定的开源协议），填写 package.json 和 Cargo.toml 的 license/author/description 字段。

🟠 High 级问题（11 项）

High H-1. read_pdf_bytes / open_path / get_pdf_hash 无路径校验
文件：src-tauri/src/lib.rs:76-98
三个命令接受任意路径字符串，无任何校验。read_pdf_bytes 可读取系统任意文件（如 ~/.ssh/id_rsa）并返回二进制；open_path 可调用系统默认程序打开任意路径/URL。若 webview 存在 XSS，攻击者可利用这些命令读取敏感文件。
校验文件扩展名为 .pdf，或维护 dialog 授权文件白名单；open_path 限制为 http/https 协议。
High H-2. 文件写入非原子操作，崩溃时数据损坏
文件：src-tauri/src/lib.rs:318-404（save_pdf_data / save_session / save_settings / save_recent_files）
所有持久化函数使用 std::fs::write 直接覆写目标文件。若写入过程中进程崩溃、断电或磁盘满，文件会被截断为部分内容，导致用户所有批注数据损坏且不可恢复。用户积累的批注和解读会话是有价值数据，不应因一次异常写入而全部丢失。
实现原子写入：先写 .tmp 临时文件，成功后 rename 覆盖目标文件。
High H-3. lookup_word 阻塞 I/O 在 async runtime 上执行
文件：src-tauri/src/lib.rs:544-547
lookup_word 是 async Tauri command，但直接调用同步的 SQLite 查询（持有 Mutex 执行阻塞 prepare + query_row）。在 Tauri 2 的 tokio 运行时上，阻塞 I/O 会卡住工作线程，影响其他异步任务调度。其他命令（read_pdf_bytes 等）都正确使用了 spawn_blocking，唯独此处遗漏。
用 tauri::async_runtime::spawn_blocking 包裹 dictionary::lookup_word 调用。
High H-4. handleAnnotationDelete 手动写盘与防抖 effect 竞态
文件：src/hooks/usePersistence.ts:496-502
handleAnnotationDelete 在删除解读标注时手动执行 loadPdfData → savePdfData（读-改-写），同时防抖 effect（行 182-215）也会在 500ms 后写盘。若防抖 effect 在手动写入之后执行且内存 state 尚未更新，删除操作会被覆盖（被删除的标注重新写回磁盘）。
统一以防抖 effect 为唯一写入入口，handleAnnotationDelete 只更新内存 state。
High H-5. 分屏视图两个 PdfViewer 共享同一 annotations 数组，标注跨文件错显
文件：src/App.tsx:374-430
usePersistence 只加载 activeTab 的 annotations，secondary tab 的 annotations 从未加载。两个 PdfViewer 接收同一个 annotations 数组，而 Annotation 无 fileHash 字段，PdfAnnotations 只按 position.page 过滤。若两个 PDF 有相同页码，primary PDF 的标注会错误显示在 secondary PDF 上。
为 Annotation 增加 fileHash 字段并同时按 fileHash + page 过滤；或分屏时维护独立的 annotations 数组。
High H-6. 关闭非活跃 Tab 时运行中的流式请求未 abort，sessions 未清理
文件：src/App.tsx:231-246
handleCloseTab 仅清理 stashes，被关 Tab 关联的正在运行的 LLM 流式请求不会被 abort（继续消耗 API token），sessions 保留在全局 state 中。关闭 secondary tab（非 active）时 annotations 也残留内存。
在 onClose 中找到该 tab 关联的所有正在流式的 session 并 abort，清理对应的 sessions 引用和 annotations。
High H-7. CSP connect-src 允许 http:，API Key 可被发送到任意 HTTP 端点
文件：src-tauri/tauri.conf.json:26
CSP connect-src 'self' http: https: 允许连接任意 HTTP/HTTPS 端点。LLM API 请求从前端 webview 直接发出，API Key 在 Authorization header 中。若发生 XSS，攻击者可将 API Key 发往任意 HTTP 服务器。
将 http: 限制为 http://localhost:* http://127.0.0.1:*（支持本地 LLM 如 Ollama），或考虑将 LLM 请求代理到 Rust 后端。
High H-8. 无崩溃监控 / 错误上报系统
文件：全项目
完全无 Sentry、Bugsnag 或自定义 telemetry。前端仅有 MarkdownErrorBoundary 局部错误边界，无全局 ErrorBoundary。Rust 端 app.run(|_, _| {}) 的 run 回调为空，未处理运行时异常。Release 构建中 tauri-plugin-log 未初始化（仅 debug 模式启用），生产环境无任何日志。用户遇到问题时无法提供日志用于排查。
集成 Sentry（@sentry/react + Rust crate）；添加 App 级 React ErrorBoundary；release 中初始化 tauri-plugin-log（Warn 级别）输出到日志文件。
High H-9. 核心业务逻辑 Hook（usePersistence.ts，547 行）零直接测试
文件：src/hooks/usePersistence.ts
这是整个应用的核心编排层，管理 annotations/sessions/stash/LLM 流式/持久化/abort 控制，约 547 行代码，无任何直接单元测试。仅通过 App.test.tsx 间接覆盖部分路径。C-1（StrictMode 双发）正是因缺少直接测试而未被发现。估算整体语句覆盖率约 55-65%，分支覆盖率约 45-55%，远低于商业级 80%+ 标准。
为 usePersistence 编写直接的单元测试，覆盖流式启动/中止、持久化防抖、删除清理、多文件 session 联动等核心路径。
High H-10. API Key 明文存储在 JSON 文件中
文件：src-tauri/src/lib.rs:216-222（LlmConfig.api_key）
API Key 以明文 JSON 存储在 AppData/SpecReader/settings.json 中。任何有文件系统访问权限的程序或用户都可读取。已从 localStorage 迁移到后端文件（改进），但仍未达到 PRD 中"系统 Keychain 或安全存储"的要求。
使用 keyring crate 或 tauri-plugin-keyring，将 API Key 存入系统钥匙串（macOS Keychain / Windows Credential Manager / Linux Secret Service）。
High H-11. CI 缺少 type-check / lint / build 验证步骤
文件：.github/workflows/ci.yml
CI 缺少 tsc --noEmit 类型检查、eslint 代码规范检查、npm run build 构建验证。类型错误只能在本地 build 时发现。项目甚至未配置 eslint 和 prettier。缺少 npm audit 和 cargo audit 安全扫描。
添加 eslint + prettier 配置；CI 中增加 type-check、lint、build、audit 步骤；添加覆盖率阈值（lines 80%）。

📊 各维度专项评估

架构设计 ⭐⭐⭐⭐ A-

Tauri 命令与纯逻辑解耦出色，service 层职责清晰，7 个自定义 Hook 拆分合理。PDF 渲染使用 IntersectionObserver 懒加载。主要不足：useStreaming 未抽取、App.css 未按组件拆分、PdfViewer 1110 行偏大。

安全性 ⭐⭐⭐ C+

Markdown 渲染使用 rehypeSanitize + 自定义 schema，XSS 防护良好。但存在路径穿越漏洞（C-2）、无路径校验（H-1）、CSP connect-src 过宽（H-7）、API Key 明文存储（H-10）。词典下载 URL 硬编码无 SSRF 风险，SQLite 参数化查询无注入风险。

数据可靠性 ⭐⭐⭐ C+

批注按 PDF SHA-256 hash 持久化，重命名不影响恢复。但文件写入非原子（H-2），崩溃时数据损坏风险；无并发写入保护（M-1）；handleAnnotationDelete 与防抖 effect 竞态（H-4）。序列化往返测试出色。

性能 ⭐⭐⭐⭐ B+
