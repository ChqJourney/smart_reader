3. 已验证的优势（值得保留）

安全基础到位

  • API Key 存系统钥匙串，磁盘 settings.json 仅留空占位；钥匙串不可用时拒绝保存而非回退明文（secure_storage.rs、lib.rs）。
  • PDF 读取命令强制授权白名单 + 扩展名校验（validate_pdf_access）。
  • 会话 id 路径穿越防护 + 单测覆盖（validate_session_id）。
  • 文件写入采用 tmp + rename 原子写 且同文件串行锁。
  • LLM 输出 Markdown 经 rehype-raw → rehype-sanitize 净化，阻断脚本注入。

工程化与质量门禁

  • CI 含 type-check / lint / build / npm audit / cargo test / clippy -D warnings / cargo audit / e2e，属小型团队中的高标准。
  • 184 前端单测 + 37 Rust 单测全绿；后端将纯逻辑与 AppHandle 解耦，可测性好。
  • 词典下载支持 Range 断点续传、zip 炸弹防护、schema 校验。
  • 旧 identifier → 新 identifier 数据目录自动迁移（paths.rs）。
  • 旧明文 Key 自动迁移至钥匙串并清空磁盘字段。

4. 问题清单（按严重度）

S1 · API Key 驻留前端内存，LLM 请求由 WebView 直连
High
位置：src/services/llm.ts:38-50 · src/services/settings.ts:98-112 · src-tauri/src/lib.rs:870-889
描述：后端从钥匙串读出 API Key 经 IPC 返回前端，Key 长期驻留 JS 内存；fetch 的 Authorization: Bearer 直接在前端构造。当前 CSP connect-src 允许任意 https:（见 S3），一旦 WebView 存在任何注入面（含被篡改/恶意的 LLM 端点通过未净化路径），Key 可被读取外泄。
影响：违背“密钥不出后端”的商业级安全基线；调试构建下可通过 DevTools 直接读取 Key。
建议：将 LLM 调用代理到 Rust 后端（新增 Tauri 命令透传 SSE），Key 仅存于 Rust；此后 CSP 可收紧为 'self' http://localhost:*，彻底消除 S1+S3 的耦合风险。这是商业级最关键架构改进。
D1 · 无代码签名（macOS ad-hoc / Windows 未签名）
High · 发布阻断
位置：src-tauri/tauri.conf.json:43 · .github/workflows/cd.yml
描述：macOS signingIdentity: "-"（ad-hoc），Windows 仅产出未签名 exe。正式发布时 Gatekeeper 会报“应用已损坏”，Windows SmartScreen 报“未知发布者”。
影响：商业公开发布的硬门槛，直接决定用户能否正常安装运行。
建议：采购 Apple Developer ID 并完成 notarization；Windows 采用 Authenticode 签名（CI 注入证书）。此为对外发布前必须项。
R1 · pdf.js 文档未 destroy，切换/关闭 PDF 内存泄漏
High · 可靠性
位置：src/components/PdfViewer.tsx:280-333（load 的 cleanup 仅置 isCancelled，未 pdf.destroy()）
描述：loadPdf 在 filePath 变化时直接替换 pdf 状态，旧 PDFDocumentProxy 从未调用 .destroy()。pdf.js 会保留已解析页面、文本与 worker 任务。
影响：桌面应用常开数小时、连续打开多个大 PDF，解析数据持续堆积，最终内存膨胀甚至卡顿/崩溃。对“常驻桌面工具”是核心可靠性缺陷。
建议：在 load effect 的 cleanup 中 pdf?.destroy()；切换前先销毁旧文档再置空。
P1 · 大文档性能：全页遍历 + 搜索无防抖
Medium · 性能
位置：PdfViewer.tsx:542-563（视口预计算）· PdfViewer.tsx:629-682（搜索）
描述：loadViewports 对每一页调用 getPage(i)，且 viewportsReady 不就绪前禁止跳页；build() 对每页 getTextContent() 且在每次按键（searchQuery 变化）重跑全量扫描，无防抖。
影响：对 PRD 明确目标的 100MB / 200 页文档，打开就绪与搜索会明显超阈值（<5s、流畅），主线程卡顿。
建议：视口尺寸改为按需/懒计算或仅计算可视页；搜索 query 增加 250–300ms 防抖；大文档文本抽取可放到 Web Worker。
S2 · 离线词典下载无完整性校验
Medium
位置：src-tauri/src/dictionary.rs:32,37（EXPECTED_DICT_ZIP_SHA256 / EXPECTED_DICT_SQLITE_SHA256 均为空串）
描述：下载后仅做 schema 校验（是否存在 stardict 表），未校验 SHA-256。代码已预留校验入口但常量留空。
影响：供应链/MITM 风险下可能加载被篡改的 sqlite 并由 rusqlite 打开查询（潜在利用 CVE）。下载本身走 HTTPS，风险中等。
建议：发布时计算并固化两个 SHA-256；校验失败删除并重试/提示。
S3 · CSP connect-src 允许任意 https:
Medium
位置：src-tauri/tauri.conf.json:26
描述：因 LLM 请求由前端直发（S1），CSP 必须放行任意 https，属于“为兼容用户自定义 Base URL 而放宽”的妥协。
建议：完成 S1（LLM 代理到后端）后，将 CSP 收紧为 'self' http://localhost:*，WebView 不再直连外网，大幅收缩 XSS 外泄面。
D2 · 未产出 macOS / Linux 安装包
Medium · 分发
位置：.github/workflows/cd.yml（仅 Windows runner）
描述：PRD 承诺支持 macOS 12+（Intel + Apple Silicon），但 CD 仅构建 Windows。即便本地可打 dmg，也无签名/notarize 流水线。
建议：补齐 macOS runner（含签名+notarize）与可选 Linux 包，与 D1 一并纳入发布流水线。
S4 · capabilities 授予未使用的 shell:default
Low
位置：src-tauri/capabilities/default.json:10
描述：应用通过 open crate 直接打开路径，并未使用 tauri-plugin-shell 的执行能力，但 capability 仍授予 shell:default。
建议：移除 shell:default 与 tauri-plugin-shell 依赖，缩小攻击面。
S5 · load/save_pdf_data 未调用 validate_pdf_access
Low
位置：src-tauri/src/lib.rs:798-827
描述：read_pdf_bytes/get_pdf_hash 强制白名单校验，但 load_pdf_data/save_pdf_data 未校验（写入仅落在 AppData 目录、按 hash 命名，实际风险低）。这与既有安全模型不一致。
建议：在这两个命令入口也调用 validate_pdf_access，保持纵深防御一致。
S6 · 日志脱敏为“尽力而为”
Low
位置：src/services/logs.ts:22-34
描述：仅匹配 sk-[A-Za-z0-9_-]{20,} 与 Bearer ...；非 sk- 格式（如某些厂商前缀、Azure 长密钥）可能漏脱敏。当前 Key 本身未被记录（好），但若后续有人记录请求体则可能泄露。
建议：扩展规则覆盖常见前缀，并对“任何包含 secret/key/token 的日志”做更保守处理。
R2 · saveSettings/loadSettings 静默吞错
Low
位置：src/services/settings.ts:124-130, 108-112
描述：钥匙串写入失败时被 catch 吞掉，用户无感知；内存中的 Key 仍可用但重启后配置丢失。
建议：将错误回传给调用方并在 UI 提示“配置保存失败”。
S7 · open_path 命令已无前端调用方（死代码）
Low
位置：src-tauri/src/lib.rs:244-256
描述：历史审核已确认该命令无调用方。保留它扩大了 IPC 攻击面。
建议：移除命令或在 capability 中严格限制协议。
C1 · Mutex 用 .unwrap()，panic=abort 下 poison 即进程退出
Low
位置：src-tauri/src/lib.rs:52-61
描述：authorize_path/is_path_allowed 对锁 .unwrap()；release 用 panic="abort"，一旦锁被 poison 整个进程中止。
建议：改为 unwrap_or_else(|e| e.into_inner())。
C2 · 写锁注册表与哈希缓存长期不下发
Low
位置：src-tauri/src/lib.rs:600-612（ATOMIC_WRITE_LOCKS）· AppState.pdf_hash_cache
描述：两者均为进程级长期增长结构，随打开文件数缓慢膨胀。对单用户桌面影响有限。
建议：写锁用完后移除；哈希缓存加容量上限/LRU。
P2 · read_pdf_bytes 整文件读入内存 + IPC ArrayBuffer
Low
位置：src-tauri/src/lib.rs:310-325 · PdfViewer.tsx:302-308
描述：整个 PDF 一次性读入 Vec<u8> 并经 IPC 传为 ArrayBuffer 交给 pdf.js，100MB+ 文件内存峰值高。
建议：对大文件可改流式分块或自定义 transport；至少给出内存上限与降级提示。
D3/D4/D5 · 体验与分发细节
Low
位置：设置/关于页、e2e/、vite.config.ts
  • D3 设置保存失败无反馈；缺“关于/许可”页（C-5 仅加 LICENSE 文件）。
  • D4 E2E 仅覆盖布局与页码跳转，缺“打开→选中→翻译/解读→持久化→重开恢复”黄金路径端到端覆盖。
  • D5 测试覆盖率阈值未强制（vite 未设 thresholds）；部分测试有 act() 警告。
  • R3 旧明文 Key 迁移中若钥匙串写入失败，明文会残留磁盘（边角场景）。

5. 商业发布阻断项（必须修复）

发布前 Must-Fix（4 + 1 项）

  • D1 代码签名：macOS Developer ID + notarization，Windows Authenticode。
  • S1 LLM 代理到 Rust 后端，API Key 不进入 WebView 内存。
  • R1 pdf.js 文档 destroy()，消除长会话内存泄漏。
  • P1 大文档（100MB/200 页）性能：视口懒计算 + 搜索防抖。

7. 结论

SpecReader AI 在 安全基础、原子持久化、密钥管理、净化渲染与 CI 质量门禁 上已达到甚至超过同体量团队的平均水平，2026-07-11 历史审核中的 Critical / High 项也基本落实。代码可读、结构清晰、测试可信。

但以“商业级公开发布”标准衡量，仍存在若干产品化阻断项：未签名导致用户无法顺畅安装、API Key 仍在前端、PDF 文档内存泄漏与大文档性能不满足 PRD 指标。修复 P0 五项后即可视为达到商业发布门槛；其余为可使产品更稳健、更专业的收敛项。

综合评分 72 / 100（B−）：高质量 MVP 基础，尚未完成产品化。
